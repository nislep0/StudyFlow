# StudyFlow – сервіс для керування навчальними завданнями

## Опис ідеї

StudyFlow – це веб-застосунок для студентів, який допомагає керувати навчальними завданнями, дедлайнами та прогресом виконання.

Основні можливості:
- Створення та редагування списку навчальних дисциплін.
- Додавання завдань з дедлайнами.
- Відстеження статусу завдання: "заплановано", "в процесі", "виконано".
- Перегляд дашборду з найближчими дедлайнами.
- Базова статистика: кількість виконаних/прострочених завдань за період.

Цільова аудиторія – студенти, які хочуть централізовано планувати навчальні завдання та вчасно їх виконувати.

## Цілі проєкту

- Надати простий веб-інструмент для керування навчальними задачами.
- Продемонструвати повний цикл розробки ПЗ:
  - проектування структури застосунку;
  - робота з базою даних;
  - покриття коду тестами;
  - розгортання та налаштування CI/CD;
  - аналіз продуктивності та оптимізація.

## Основний функціонал

1. **Користувачі**
   - Реєстрація/логін (мінімально – email + пароль).
   - Зберігання налаштувань користувача.

2. **Курси**
   - CRUD для курсів.
   - Прив’язка курсів до користувача.

3. **Завдання**
   - CRUD для завдань.
   - Поля завдання: назва, опис, курс, дедлайн, статус, пріоритет.
   - Фільтрація завдань.

4. **Дашборд**
   - Список найближчих дедлайнів.
   - Простий індикатор навантаження.

## Технологічний стек (попередній)

- **Frontend**: React + TypeScript
- **Backend**: Node.js + TypeScript
- **База даних**: PostgreSQL
- **ORM/DB-інструмент**: Prisma або TypeORM
- **Тести**: Jest / Vitest, Playwright / Cypress для E2E
- **Інструменти якості коду**: ESLint, Prettier
- **CI/CD**: GitHub Actions
- **Розгортання**: Railway/Render/Vercel/Netlify (буде уточнено при виконанні ЛР №6)

## План виконання лабораторних робіт

- **ЛР №0** – вибір ідеї, створення репозиторію, опис у README.
- **ЛР №1** – налаштування структури проєкту, форматтер, лінтер, git-hooks.
- **ЛР №2** – проектування архітектури застосунку (компонентна діаграма, ER-діаграма, опис сценаріїв).
- **ЛР №3** – інтерактивний прототип з використанням статичних даних.
- **ЛР №4** – інтеграція з БД, заміна статичних даних на реальні.
- **ЛР №5** – покриття модулів тестами.
- **ЛР №6** – CI/CD, розгортання сервісу та доступ з інтернету.
- **ЛР №7** – аналіз продуктивності.
- **ЛР №8** – виділення “вразливого” модуля в окрему бібліотеку, рефакторинг, перевід на TypeScript.
